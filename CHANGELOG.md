# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- `init` skill now features an interactive interview mode ÔÇö instead of copying template files full of placeholders, it walks the user through grouped question phases and generates populated config files from their answers
- `init` skill now supports **update mode** ÔÇö re-running `/project-workflows:init` on an already-configured project shows a status dashboard and lets the user update individual sections, add new repositories, or configure skills they previously skipped
- `init` skill detects legacy (non-interactive) config files and offers to replace or update them
- Config files generated by the interactive init now include a `<!-- pw-version: 1.1.0 -->` stamp on the first line, enabling update mode detection and future template version migration
- Template files in `skills/init/templates/` now include `<!-- pw-version: 1.1.0 -->` stamps

### Fixed

- Aligned env var names across all skills: renamed `REPO_API_TOKEN` Ôćĺ `API_TOKEN_ENV_VAR` and `REVIEW_API_TOKEN` Ôćĺ `REVIEW_TOKEN_ENV_VAR` so that `.env.example`, `init` skill validation, and all operational skill prerequisites use consistent names
- `init` Step 4 guidance now lists `REVIEW_TOKEN_ENV_VAR` alongside `API_TOKEN_ENV_VAR`
- `.env.example` comment for `REVIEW_TOKEN_ENV_VAR` now clarifies it is consumed by the `code-review` skill
- Fixed misleading "Installing a specific version" instructions in README that caused a `Marketplace file not found` error when using a relative local path; instructions now require an absolute path

## [1.0.0] - 2026-03-25

### Added

- `init` skill ÔÇö scaffolds project configuration files (PROJECT.md, REVIEW-CRITERIA.md, TEST-MATRIX.md, PRD-MANIFEST.md, .env.example)
- `code-review` skill ÔÇö reviews all open change requests across configured repositories using a coordinator + sub-agent pattern
- `development` skill ÔÇö implements features, bug fixes, or tasks from issues via exploration, implementation, and review sub-agents
- `issue-creation` skill ÔÇö creates well-structured issues from problem descriptions or feature requests
- `testing-prd` skill ÔÇö runs integration tests generated dynamically from PRDs
- `testing-static` skill ÔÇö runs integration tests using a static test matrix
- `gitlab-api` reference skill ÔÇö GitLab REST API operations (MRs, issues, notes, branches)
- `github-api` reference skill ÔÇö GitHub REST API operations (PRs, issues, reviews, branches)
- `gitea-api` reference skill ÔÇö Gitea REST API operations (PRs, issues, comments, branches)
- `bug-fix` shared sub-agent ÔÇö reusable bug-fix workflow
- GitLab CI pipeline ÔÇö validates plugin structure and skill frontmatter on changes
- GitHub Actions workflow ÔÇö mirrors GitLab CI validation for GitHub-hosted mirrors
